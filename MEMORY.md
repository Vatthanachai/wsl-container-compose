# MEMORY.md

Load-bearing facts and past incidents worth checking before you act in this repo. Not a full history — see `obsidian/wsl-container-compose/Progress.md` for that.

- **The real `IContainerRuntime` adapter doesn't exist.** `Cli`'s `LoadProject` wires up `NotImplementedContainerRuntime`. Don't be surprised that `up`/`down`/etc. throw `NotImplementedException` when run for real — that's expected today, not a bug to fix in passing.
- **`dotnet pack` on `WslContainerCompose.Cli` fails** (`NETSDK1146`): `PackAsTool` doesn't support the project's Windows-specific TFM. This is a known, accepted constraint — releases publish a self-contained `win-x64` zip instead (see `COMMANDS.md`, `.github/workflows/release.yml`). Don't "fix" this by changing the TFM or dropping `PackAsTool` without discussing it first.
- **Release tags must match `v[0-9]+.[0-9]+.[0-9]+` exactly** (e.g. `v0.1.0`). A previous release attempt used `v.0.1.0` (stray dot after `v`), which matched the old looser tag glob and broke the version-parsing step in the workflow with `'.0.1.0' is not a valid version string`. The tag pattern was tightened afterward specifically to reject malformed tags like this.
- **`networks:` support is designed but not implemented.** The design in `Plan.md` ("Networks (provisional)") is marked provisional because it hasn't been validated against a real WSL install — in particular, whether `Container.Inspect()`'s raw payload actually contains a parseable IP address is an assumption, not a confirmed fact. Per `Progress.md`'s "Next up", this networks work is sequenced **ahead of** the real adapter, and should be built and tested against `FakeContainerRuntime` first, consistent with how every other feature here was built.
- **New orchestration/parsing logic gets built and tested against `FakeContainerRuntime` first**, real-adapter integration comes later. This isn't a stopgap being cut for lack of time — it's the deliberate pattern for this codebase (see "Testability" in `Plan.md`).
